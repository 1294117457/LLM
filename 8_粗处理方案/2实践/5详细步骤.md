## 步骤一：定义通用 Schema 并验证可行性

### 做什么

确定你的语句提纯模型的**输入输出规范**，这是后续所有工作的基础。

### 怎么做

**1.1 创建 schema 定义文件**

在项目中新建 `schema/intent_schema.json`，写入规范定义：

```json
{
  "intent_type": {
    "enum": ["query", "compare", "recommend", "action", "chat"],
    "description": "语句目的分类"
  },
  "action": {
    "type": "string",
    "description": "动宾短语，如 查询条件、对比学校"
  },
  "entities": {
    "type": "array",
    "items": {
      "type": { "enum": ["organization", "field", "person", "location", "time", "topic", "metric", "product", "other"] },
      "value": { "type": "string" }
    }
  },
  "constraints": {
    "type": "array",
    "items": "string"
  },
  "normalized_query": {
    "type": "string",
    "description": "标准化后的简洁核心表述"
  }
}
```

**1.2 手动验证 schema 可行性**

在开始批量生成数据之前，先用大模型 API 手动测试 20-30 条不同领域的句子，确认 schema 设计合理。

打开你常用的大模型对话界面（ChatGPT / 通义千问 / 任何 API），逐条测试：

```
Prompt：
请将以下用户问句提取为结构化意图 JSON，格式如下：
{
  "intent_type": "query|compare|recommend|action|chat",
  "action": "动宾短语",
  "entities": [{"type": "organization|field|person|location|time|topic|metric|product|other", "value": "值"}],
  "constraints": ["约束条件"],
  "normalized_query": "标准化简洁表述"
}
只输出 JSON。

用户问句：我想问问浙大和复旦计算机哪个好保研
```

**1.3 记录问题并调整**

把 20-30 条测试结果记录在一个表格中：

| 输入 | 大模型输出 | 是否符合预期 | 需要调整的地方 |
|------|----------|------------|-------------|
| ... | ... | 是/否 | ... |

根据结果调整 schema。

### 可能遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| 大模型给 `intent_type` 输出了 schema 外的值（如 "question"） | Prompt 约束不够强 | 在 prompt 中明确强调 "intent_type 只能是以下五个值之一"，并在 prompt 末尾加 "严格遵守格式，不要自创字段" |
| 多意图句子不知道怎么标注（如 "对比清华和北大，然后推荐一个"） | schema 不支持多意图 | 暂时取主意图（对比），把次意图放 constraints 中（如 `"constraints": ["推荐"]`）。不要为了覆盖边缘 case 把 schema 搞复杂，0.5B 模型学不好 |
| entity type 选择纠结（如 "保研" 是 topic 还是 action？） | 分类边界模糊 | 定一个规则然后保持一致。建议："保研"归 topic（它是一个主题），"申请保研"中的"申请"归 action 字段 |
| 闲聊类句子大模型输出格式不统一 | 闲聊没什么实体可提 | 统一规定闲聊输出固定模板：`{"intent_type": "chat", "action": "闲聊", "entities": [], "constraints": [], "normalized_query": "闲聊"}` |
| `normalized_query` 大模型输出风格不一致（有的像关键词堆叠，有的是完整句子） | 没有明确规范 | 在 prompt 中加示例说明 normalized_query 的风格："用空格分隔的关键词组合，如 `清华大学 计算机 保研 GPA要求`" |

### 完成标志

- [ ] schema 定义文件已创建
- [ ] 手动测试 20-30 条，覆盖至少 5 个不同领域
- [ ] 大模型输出 90%+ 符合预期
- [ ] 发现的边缘 case 有明确处理规则

# 关键调整点：

### 可能遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| API 调用报错 `rate_limit_exceeded` | 请求频率太快 | 在每次 API 调用后加 `time.sleep(1)` 或 `time.sleep(2)`，降低频率 |
| 大模型返回的不是合法 JSON，而是带了 ` ```json ``` ` 包裹 | 大模型自作主张加了 markdown 格式 | 在解析前做预处理：`content = content.strip().removeprefix("```json").removesuffix("```").strip()` |
| 生成的数据领域分布不均（某些领域生成失败率高） | 某些领域大模型不太擅长编造 | 检查各领域的生成成功率，失败率高的领域单独调整 prompt 或增加 batch 数 |
| 清洗后数据量不够（比如只剩 1500 条） | 生成质量不够 | 分析失败原因，修改 prompt 后重新生成差额部分。不要降低清洗标准 |
| 数据同质化严重（很多句子表述雷同） | temperature 不够高 / 参考 examples 太少 | 把 `temperature` 提高到 `0.95`，给每批生成不同的参考 examples，在 prompt 中加 "请使用多样化的表达方式" |
| API 费用超预期 | 生成量大 | 用 Qwen-Plus（阿里百炼）或 GPT-4o-mini（比 GPT-4 便宜 20 倍）。3000 条数据用 GPT-4o-mini 大约 ¥5-10，Qwen-Plus 新用户有免费额度 |
| 脚本中途意外中断，之前的数据丢失 | 没有做断点保存 | 每 100 条保存一次 checkpoint。重新运行时先加载 checkpoint 继续 |

### 可能遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `OutOfMemoryError: CUDA out of memory` | 显存不够 | 把 `per_device_train_batch_size` 从 4 降到 2 或 1。如果还不行，把 LoRA `r` 从 32 降到 16 |
| 训练 loss 不下降，一直在某个值震荡 | 学习率不合适 / 数据有问题 | 先排查数据：随机打印几条训练数据确认格式正确。如果数据没问题，尝试调整学习率：`1e-4` → `5e-5` 或 `2e-4` |
| 训练 loss 极速下降后又飙升 | 学习率太大，过拟合 | 降低学习率到 `5e-5`，增加 `warmup_steps=100`（在 SFTConfig 中添加） |
| eval_loss 下降后开始上升（训练 loss 还在降） | 过拟合：模型记住了训练数据但泛化差 | 减少 `num_train_epochs`（从 5 降到 3），或增加训练数据量。`load_best_model_at_end=True` 已经配了，会自动选最佳 checkpoint |
| `ValueError: Tokenizer` 相关错误 | 数据格式与 tokenizer 不兼容 | 检查 `train.json` 中每条数据的 `instruction/input/output` 字段是否都是字符串，不能有 None 或非字符串类型 |
| AutoDL 实例突然断开 | 网络波动 / 超时 | 训练脚本已配置 `save_strategy="steps"` 和 `save_steps=100`，重连后从最新 checkpoint 恢复：在 SFTConfig 中加 `resume_from_checkpoint=True` |
| 训练速度太慢 | batch_size 太小 / 数据加载慢 | 加 `dataloader_num_workers=4`（SFTConfig 中），确认用的是 GPU 而非 CPU（检查 `device_map="auto"` 是否生效） |
| 忘记关机导致持续扣费 | 人为疏忽 | 训练完成后**立即在 AutoDL 控制台关机**。建议在训练脚本最后加一行 `print("训练完成！请立即关闭服务器！")`  |

## 步骤四：合并权重并部署到 Ollama

### 做什么

把 LoRA 权重合并为完整模型，转换格式，在本地用 Ollama 部署为 HTTP 服务。

### 怎么做

**4.1 合并 LoRA 权重（本地 CPU 可执行）**

创建 `merge.py`：

```python
from transformers import AutoModelForCausalLM, AutoTokenizer
from peft import PeftModel
import torch

model_name = "Qwen/Qwen2.5-0.5B-Instruct"

print("加载基座模型...")
base_model = AutoModelForCausalLM.from_pretrained(model_name, torch_dtype=torch.float16)

print("加载 LoRA 权重...")
model = PeftModel.from_pretrained(base_model, "./universal-intent-purifier-lora")

print("合并权重...")
merged_model = model.merge_and_unload()

print("保存合并后的模型...")
merged_model.save_pretrained("./universal-intent-purifier-merged")
AutoTokenizer.from_pretrained(model_name).save_pretrained("./universal-intent-purifier-merged")

print("完成！模型已保存到 ./universal-intent-purifier-merged")
```

```bash
pip install torch transformers peft
python merge.py
# 大约需要 2-5 分钟（CPU），输出目录约 1-2 GB
```

**4.2 转换为 GGUF 格式**

**4.6 批量测试验证**

写一个测试脚本覆盖多种场景：

```python
import requests
import json

test_cases = [
    # (输入, 期望的 intent_type, 期望包含的 entity)
    ("清华计算机保研要求", "query", "清华"),
    ("iPhone和华为手机哪个好", "compare", "iPhone"),
    ("推荐一个适合学生的笔记本", "recommend", None),
    ("帮我取消订单", "action", None),
    ("哈哈你真有意思", "chat", None),
    ("北大和清华计算机哪个更好保研", "compare", "北大"),
    ("我想了解一下复旦大学经济系的推免政策和申请流程", "query", "复旦"),
    ("Python怎么读取csv文件", "query", "Python"),
]

passed = 0
failed = 0

for query, expected_intent, expected_entity in test_cases:
    resp = requests.post("http://localhost:11434/api/generate", json={
        "model": "intent-purifier",
        "prompt": query,
        "stream": False,
    })
    raw = resp.json()["response"]

    try:
        result = json.loads(raw)
        intent_ok = result.get("intent_type") == expected_intent
        entity_ok = expected_entity is None or any(
            expected_entity in e.get("value", "") for e in result.get("entities", [])
        )

        if intent_ok and entity_ok:
            passed += 1
            status = "✅"
        else:
            failed += 1
            status = "❌"

        print(f"{status} [{query}]")
        print(f"   输出: {json.dumps(result, ensure_ascii=False)}")
        if not intent_ok:
            print(f"   ⚠ intent 期望 {expected_intent}，实际 {result.get('intent_type')}")
        if not entity_ok:
            print(f"   ⚠ 未找到期望实体 {expected_entity}")

    except json.JSONDecodeError:
        failed += 1
        print(f"❌ [{query}] JSON 解析失败: {raw[:100]}")

    print()

print(f"通过: {passed}/{passed+failed}, 失败: {failed}/{passed+failed}")
```

### 可能遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `merge.py` 报 `OutOfMemoryError` | 内存不够（合并需要 ~4GB RAM） | 你有 16GB 内存应该够。如果同时跑了其他程序，关掉后再试。或者在云服务器上合并后下载合并好的模型 |
| `convert_hf_to_gguf.py` 报错 `KeyError` 或 `Unsupported model` | llama.cpp 版本太旧不支持 Qwen2.5 | `git pull` 更新 llama.cpp 到最新版。Qwen2.5 的 GGUF 支持是 2024 年下半年才加的 |
| `ollama create` 报错 | GGUF 文件损坏或格式不对 | 重新转换，试试不同的量化方案：`--outtype f16`（不量化，体积更大但兼容性最好） |
| Ollama 推理输出不是 JSON，而是自然语言 | Modelfile 中的 SYSTEM prompt 没生效 | 确认 SYSTEM 内容中明确写了"只输出JSON"。如果还不行，在调用时把 system prompt 拼进 prompt 参数中 |
| Ollama 推理输出 JSON 被截断 | `num_predict` 太小 | 把 Modelfile 中 `num_predict` 从 300 提到 500 |
| 推理速度慢（每条 > 5 秒） | CPU 推理 0.5B 模型，正常偏慢 | 0.5B 量化后 CPU 推理应在 1-3 秒。如果超过 5 秒，检查系统是否有其他高负载进程。有显卡的话 Ollama 会自动使用 GPU 加速 |
| Windows 下 Ollama 服务起不来 | 端口被占用 / 防火墙 | 检查 11434 端口是否被占：`netstat -ano | findstr 11434`。或换端口：设环境变量 `OLLAMA_HOST=0.0.0.0:11435` |

## 步骤五：搭建语义缓存系统

### 做什么

用 SQLite 搭建缓存数据库，实现硬匹配 + 软匹配的查询逻辑和 LFU 淘汰。

### 怎么做

**5.1 创建缓存数据库管理模块**

新建文件 `cache_manager.py`：

```python
import sqlite3
import json
import time
from difflib import SequenceMatcher
from pathlib import Path

DB_PATH = "intent_cache.db"

def init_db():
    """初始化数据库，创建表"""
    conn = sqlite3.connect(DB_PATH)
    conn.execute("""
        CREATE TABLE IF NOT EXISTS intent_cache (
            id INTEGER PRIMARY KEY AUTOINCREMENT,
            intent_type TEXT NOT NULL,
            action TEXT NOT NULL,
            entities_json TEXT NOT NULL,
            constraints_json TEXT NOT NULL,
            normalized_query TEXT NOT NULL,
            original_query TEXT NOT NULL,
            answer TEXT NOT NULL,
            hit_count INTEGER DEFAULT 0,
            created_at REAL DEFAULT (strftime('%s','now')),
            last_hit_at REAL
        )
    """)
    conn.execute("CREATE INDEX IF NOT EXISTS idx_intent_type ON intent_cache(intent_type)")
    conn.execute("CREATE INDEX IF NOT EXISTS idx_hit_count ON intent_cache(hit_count)")
    conn.commit()
    conn.close()
    print(f"数据库已初始化: {DB_PATH}")


def search_cache(purified_intent: dict, similarity_threshold: float = 0.80) -> tuple:
    """
    搜索缓存，返回 (匹配结果 dict | None, 覆盖率分数 float)
    """
    conn = sqlite3.connect(DB_PATH)
    conn.row_factory = sqlite3.Row
    cursor = conn.cursor()

    cursor.execute(
        "SELECT * FROM intent_cache WHERE intent_type = ?",
        (purified_intent.get("intent_type", ""),)
    )
    candidates = cursor.fetchall()

    if not candidates:
        conn.close()
        return None, 0.0

    best_match = None
    best_score = 0.0

    input_entities = purified_intent.get("entities", [])
    input_entity_set = {(e["type"], e["value"]) for e in input_entities}
    input_query = purified_intent.get("normalized_query", "")

    for row in candidates:
        cache_entities = json.loads(row["entities_json"])
        cache_entity_set = {(e["type"], e["value"]) for e in cache_entities}

        # 硬匹配：entity 重叠度
        if cache_entity_set and input_entity_set:
            overlap = len(cache_entity_set & input_entity_set) / max(len(cache_entity_set), len(input_entity_set))
        elif not cache_entity_set and not input_entity_set:
            overlap = 1.0
        else:
            overlap = 0.0

        if overlap < 0.8:
            continue

        # 软匹配：normalized_query 文本相似度
        text_sim = SequenceMatcher(None, row["normalized_query"], input_query).ratio()

        score = overlap * 0.6 + text_sim * 0.4

        if score > best_score:
            best_score = score
            best_match = dict(row)

    conn.close()

    if best_match and best_score >= similarity_threshold:
        return {
            "id": best_match["id"],
            "answer": best_match["answer"],
            "original_query": best_match["original_query"],
            "hit_count": best_match["hit_count"],
            "coverage": round(best_score * 100, 1),
        }, best_score
    else:
        return None, best_score


def save_to_cache(purified_intent: dict, original_query: str, answer: str):
    """写入新的缓存记录"""
    conn = sqlite3.connect(DB_PATH)
    conn.execute("""
        INSERT INTO intent_cache (intent_type, action, entities_json, constraints_json, normalized_query, original_query, answer)
        VALUES (?, ?, ?, ?, ?, ?, ?)
    """, (
        purified_intent.get("intent_type", ""),
        purified_intent.get("action", ""),
        json.dumps(purified_intent.get("entities", []), ensure_ascii=False),
        json.dumps(purified_intent.get("constraints", []), ensure_ascii=False),
        purified_intent.get("normalized_query", ""),
        original_query,
        answer,
    ))
    conn.commit()
    conn.close()

    evict_lfu()


def update_hit(cache_id: int):
    """更新命中次数和最后命中时间"""
    conn = sqlite3.connect(DB_PATH)
    conn.execute("""
        UPDATE intent_cache SET hit_count = hit_count + 1, last_hit_at = strftime('%s','now')
        WHERE id = ?
    """, (cache_id,))
    conn.commit()
    conn.close()


def evict_lfu(max_size: int = 100):
    """LFU 淘汰"""
    conn = sqlite3.connect(DB_PATH)
    cursor = conn.cursor()
    cursor.execute("SELECT COUNT(*) FROM intent_cache")
    count = cursor.fetchone()[0]

    if count > max_size:
        delete_count = count - max_size
        cursor.execute("""
            DELETE FROM intent_cache WHERE id IN (
                SELECT id FROM intent_cache ORDER BY hit_count ASC, last_hit_at ASC LIMIT ?
            )
        """, (delete_count,))
        conn.commit()

    conn.close()


def get_cache_stats():
    """获取缓存统计信息"""
    conn = sqlite3.connect(DB_PATH)
    cursor = conn.cursor()
    cursor.execute("SELECT COUNT(*) FROM intent_cache")
    total = cursor.fetchone()[0]
    cursor.execute("SELECT SUM(hit_count) FROM intent_cache")
    total_hits = cursor.fetchone()[0] or 0
    cursor.execute("SELECT AVG(hit_count) FROM intent_cache")
    avg_hits = cursor.fetchone()[0] or 0
    conn.close()
    return {"total_records": total, "total_hits": total_hits, "avg_hits_per_record": round(avg_hits, 2)}
```

### 可能遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| `SequenceMatcher` 对短文本相似度偏高（如 "查GPA" 和 "查流程" 得分 0.7+） | SequenceMatcher 是字符级匹配，短字符串容易高分 | 对于 normalized_query 很短的情况（< 10 字），把 `similarity_threshold` 提高到 0.90；或在 entity 完全匹配的前提下适当降低软匹配权重 |
| SQLite 并发写入报 `database is locked` | SQLite 不支持并发写 | 开发阶段问题不大（单线程）。生产环境改用 Redis 或在写操作上加 `threading.Lock()` |
| 缓存表越来越大，查询变慢 | 没有及时淘汰 | `evict_lfu` 已在每次 `save_to_cache` 中自动调用。如果还是慢，给 `entities_json` 字段加全文索引或缩小 `max_size` |
| 硬匹配过于严格，很多相似查询无法命中（如 "清华" vs "清华大学"） | entity value 不完全一致 | 在提纯模型中统一规范：训练数据中统一用全称（"清华大学"而非"清华"）。或在匹配时做模糊匹配：`if value_a in value_b or value_b in value_a` |
| 缓存冷启动：刚开始没有数据，全部走 LLM | 正常现象 | 这就是你设计的"自举"机制——初期全部走 LLM 生成，答案自动写入缓存，后续逐渐有命中。可以预热：用常见问题列表批量填充缓存 |

## 步骤六：LangGraph 编排集成

### 做什么

用 LangGraph 把提纯模型、缓存系统、大模型生成串联成完整的端到端流程。

### 怎么做

**6.1 确定技术方案**

你的 LangGraph 项目是 TypeScript 还是 Python？

- **如果是 TypeScript（LangGraph.js）**：缓存模块需要包一层 HTTP API（用 Python FastAPI），LangGraph 通过 HTTP 调用
- **如果是 Python（LangGraph Python SDK）**：可以直接导入 `cache_manager.py`

两种方案选一个。以下以"缓存包一层 API"为例（通用性更好）：

**6.2 为缓存创建 API 服务**

```python
# cache_api.py
from fastapi import FastAPI
from pydantic import BaseModel
from cache_manager import init_db, search_cache, save_to_cache, update_hit, get_cache_stats

app = FastAPI()

class SearchRequest(BaseModel):
    purified_intent: dict

class SaveRequest(BaseModel):
    purified_intent: dict
    original_query: str
    answer: str

@app.on_event("startup")
def startup():
    init_db()

@app.post("/cache/search")
def api_search(req: SearchRequest):
    result, score = search_cache(req.purified_intent)
    if result:
        update_hit(result["id"])
    return {"hit": result is not None, "result": result, "score": score}

@app.post("/cache/save")
def api_save(req: SaveRequest):
    save_to_cache(req.purified_intent, req.original_query, req.answer)
    return {"status": "ok"}

@app.get("/cache/stats")
def api_stats():
    return get_cache_stats()
```

**6.3 LangGraph 中的节点实现**

在你的 LangGraph 项目中，每个节点调用对应服务：

```
purifyIntent 节点  →  POST http://localhost:11434/api/generate （Ollama）
searchCache 节点   →  POST http://localhost:8100/cache/search
generateAnswer 节点 → 你现有的大模型调用逻辑
saveToCache 节点   →  POST http://localhost:8100/cache/save
```

### 可能遇到的问题

| 问题 | 原因 | 解决方案 |
|------|------|---------|
| LangGraph 调用 Ollama 超时 | Ollama 第一次推理需要加载模型，冷启动慢 | 在服务启动后先发一条预热请求：`curl http://localhost:11434/api/generate -d '{"model":"intent-purifier","prompt":"test","stream":false}'` |
| 多个服务端口冲突 | 同时跑 Ollama + 缓存 API + LangGraph | 确认每个服务端口不同。Ollama 默认 11434，缓存 API 用 8100，LangGraph 应用用另一个端口 |
| 缓存 API 和 LangGraph 之间的数据格式不匹配 | JSON 字段名不一致 | 统一使用 schema 定义文件中的字段名，用 Pydantic（Python）或 Zod（TypeScript）做校验 |
| 兜底大模型调用增加了整体延迟 | 提纯失败 → 等兜底 → 再走后续流程 | 设置兜底的超时时间（如 5 秒）。如果兜底也失败，直接用原始 query 走大模型生成（跳过缓存） |
| 缓存写入和读取不一致 | 多进程/多请求并发 | 开发阶段不用管。生产环境把 SQLite 换成 Redis，天然支持并发 |

**7.2 定期分析数据**

每周看一次核心指标：

```python
# analyze.py
import json

with open("request_logs.jsonl", "r") as f:
    logs = [json.loads(line) for line in f]

total = len(logs)
cache_hits = sum(1 for l in logs if l["cache_hit"])
purify_failures = sum(1 for l in logs if not l["purify_success"])
fallbacks = sum(1 for l in logs if l["fallback_used"])
total_tokens = sum(l.get("tokens_used", 0) for l in logs)

print(f"总请求数: {total}")
print(f"缓存命中率: {cache_hits/total*100:.1f}%")
print(f"提纯成功率: {(total-purify_failures)/total*100:.1f}%")
print(f"兜底使用率: {fallbacks/total*100:.1f}%")
print(f"总 token 消耗: {total_tokens}")
print(f"平均每次 token: {total_tokens/total:.0f}")
```

**7.3 根据数据迭代**

| 发现的问题 | 对应操作 |
|-----------|---------|
| 提纯成功率 < 95% | 收集失败 case → 加入训练数据 → 重新训练 |
| 缓存命中率 < 20% | 要么用户问题太分散（正常），要么硬匹配阈值太高 → 尝试降低 entity_overlap 阈值到 0.6 |
| 用户频繁拒绝缓存答案 | 缓存答案过时或覆盖率判断不准 → 降低 similarity_threshold 或给缓存加 TTL 过期时间 |
| 兜底率 > 10% | 小模型泛化不够 → 增加训练数据 → 考虑升级到 1.5B |
